什么是Action

Action描述的是一个动作。大致形式如下：

这个type标识动作是什么，reducer根据这个type决定要不要处理。payload是携带的数据。比如说我们描述提交了一个Form，那么这个type就是“提交Form”，而payload就是Form中用户填写的数据。

action的实现中分两类。一类是同步Action，一类是异步Action。

  • 同步Action：没有网络请求。动作发生了马上就结束，不存在这个动作正在进行中这个过程。
  • 异步Action：一般为网络请求。一个动作发生了，会分为动作发生了、动作进行中、动作结束这三个过程。

同步Action

同步Action比较基础，dispatch action, reducer直接返回新的state，更新UI。

举个例子，react_start_kit 项目，我们点击Increase number，调用action creator的increase函数生成一个action，然后dispatch action，这时候reducer接收到action，执行state+1，返回新的state，更新UI，这就是一个同步action。

在view中，我们通过调用disptatch分发这个action。在src/pages/HelloWorldPage.js里可以找到类似的话

上面调用了increase()函数。这是一个action的creator。在src/action_creators/count.js.”中我们定义了自己的action。同步的Action只返回一个新的对象。因为我们只是让它增1，所以没有设置payload。你可以尝试这加一个payload，实现增加任意的数字。

reducer

如果我们要开发一个异步的引用，应该怎么处理呢？

举个例子，异步网络请求，首先有发起请求的时刻，还有请求成功或者请求失败的时刻。这些时刻，我们可能需要更新应用的状态，也就是state，比如显示加载中，加载完成，加载失败。既然要更新state，就需要dispatch action，reducer返回新的state。所以我们需要为发起请求，请求成功，请求失败定义action。

我们可以把这些action统一为一个action，内部用status区分，不过redux建议定义为不同的action，比如POSTS_REQUEST，POSTS_FAILURE，POSTS_SUCCESS。

写到这里，你可能会问，异步在哪呢，定义的这些Action和上面的Increase number的Action还是一样啊。确实一样，这些都是同步Action。

异步Action

接下来要说的异步Action就会把上面的同步Action和网络请求结合起来。

上面我们写的action creator的函数都是返回一个action对象，不能完成异步网络请求，接下来我们引入 Redux Thunk middleware，用来扩展action creator的功能。

解释一下middleware，node服务端框架常用，就是嵌入在框架接收请求到产生响应过程之中的代码。简单说middleware就是帮我们完成某些工作，并且可以链式组合多个独立的middleware。

Redux middleware类似，它提供的是位于action被发起之后，到达reducer之前的扩展。

引入Redux Thunk middleware之后，action creator不仅可以返回action对象，还可以返回函数，当action creator返回函数时，这个函数会被Redux Thunk middleware执行。

既然返回的是函数，我们就可以在函数里面做需要的工作，比如异步网络请求，并且我们可以在这个函数里面dispatch其他的action，既可以是上面的同步的action，也可以是其他异步的action，是不是快连起来了。

我们在返回的函数里面执行了fetch异步网络请求，请求之前，先dispatch一个REQUEST_POSTS的action，请求完成的时候，dispatch一个RECEIVE_POSTS的action，请求失败的时候，dispatch一个POSTS_FAILURE的action，reducer会分别处理这些action，返回相应的新的status，更新UI。

解释一下fetch，它是替代 XMLHttpRequest 用来发送网络请求的非常新的 API。由于目前大多数浏览器原生还不支持它，建议你使用 isomorphic-fetch 库。它和jquery的async相比，异步任务采用promise链式调用，减少嵌套，代码清晰。

如何引入 Redux Thunk middleware

5. Redux基础知识

Feb 28th, 2016 12:53 am

Redux不像MVC，给人的感觉更像消息系统。整个的感觉是这样的：

整个应用有一个State，这个State表明了所有的状态，并且只有一个Store存储这个。其余的操作都是action。通过action_creator创建出来一个action，然后dispatch（分发）这个action。然后所有的reducer根据action的type来决定要不要处理这个action。如果要处理，就返回一个新的State。然后Store会根据这个新的State更新State树。然后重新渲染页面。

整个过程看起来是这样的： 上面说了一些名词，下面说说他们的意思。

  • State：状态。就是那些可以变的东西。比如说用户的行为、随着时间变化的等。与React中的状态一致。它是一个简单的Object
  • Action：行为。用来描述已经发生的事件。意思上更接近做了xx事。也是一个简单的Object。推荐这样设计这个Action
    • { type: “SUBMIT_FORM”, //表明action的类型。reducer用来它决定要不要响应这个action payload: { //Action搭载的数据 username: “test”, password: “password” } }
  • Dispatch：分发。它其实是一个函数，dispatch是store的一个函数。用来分发action
  • Reducer: 处理器。它是将根据Action中的type来决定是否响应，如果响应，会根据当前的State和Action中搭载的数据。产生一个新的State。一个reducer对应这State树中一个分支。所以当前State和新的State都指的是reducer对应的那部分State
  • Store: 存储。全局只会有一个Store。存储了State树，并且提供了dispatch、getState等方法

Redux有三个原则：

  1. 单一数据源：整个应用的state被存储到一颗object tree中。并且这个object tree 存在于唯一一个store中
  2. State是只读的：唯一能够改变State的做法就是触发Action。
  3. 用纯函数来执行修改：Reducer用来描述action如何修改State树。

纯函数的意思是这样的：

  1. 里面没有网络请求，不更新数据。就是没有任何对外的影响。
  2. 不改变输入值

大致Redux总的想法就是一套消息系统，然后配合上状态机的使用。核心的事情有以下这些：

  1. 应用有哪些状态
  2. 如何在状态间切换

对于状态间的切换，它用的方式是消息。也就是Action。我们的View只需要这两个职责

  1. 根据状态决定如何显示
  2. 通知Redux发生了xx事

这是一个非常非常好的设计。干净，清晰。随着我们的应用复杂度提升，Redux的开发复杂度也能实现线性增长。而不是像MVC，会成n2甚至指数增长。

Redux项目地址：https://github.com/rackt/redux 非官方Flux实现方案，Star数超过11K，非常厉害。

另外，可以去看看Facebook官方的Flux，思想很接近。不要来说服我用官方Flux。。。 不用Flux原因有以下几个：

  1. 最重要的，David用了Redux。Flux和Redux是两套东西。后续我们的路由等等也会用Redux来做。选择了Flux意味着所有东西都要自己去尝试。没有大神腿可以抱
  2. 官方的Flux点赞数没有Redux高
  3. Redux更清晰，更干净。Flux也在越来越像Redux
  4. 鉴于我们的前端水平如此之弱，选择哪个其实都是一样的。我觉得我们是没有能力判断到底哪个好的。所以根据1、2两条原因。我们会只用Redux

一个基本原则是这样的 不管是page还是component，本质上都是一个组件。一个组件是由这样一个文件夹组成的：（以Slogan为例）

  1. 文件夹名字与组件相同。所以文件夹名为Slogan
  2. 这个文件下有这个组件需要的所有东西，比如js、css，甚至是图片等。
  3. 文件夹下有一个Slogan.js与Slogan.css。js文件负责组件的显示，资源的整合。css负责组件内的布局，它不用关心组件间该如何排列。
  4. css 的className推荐以组件名作为前缀，并且一个组件的顶级className最好和组件名相同。

最基础的组件，放在components文件夹下。有几个要点

  1. 显示的内容最好不要写死，要通过props传递。比如文字内容，最好写进props，然后在Page中初始化。这样能最大程度的重复利用
  2. 组件要分的恰当，不要太大，也不要太小。基本原则是这样的：如果一个组件会在Page或其他Component中使用，那么就独立为一个文件夹。否则这是一个内部组件，放在相应组件即可。以Banner的显示为例
  3. 暴露在外面的是Banner这个组件，一个Banner由多个BannerImage组成，Banner负责控制BannerImage的初始化、轮播
  4. BannerImage寄生在Banner组件中，放在对应的js和css中。
  5. 如果以后有组件需要使用BannerImage，那么可以将它独立为一个文件夹，然后考虑换一个更通用的名字，例如ClickableImage等。

抽象出来了一类特殊的组件，叫做page。建议名字以Page结尾。它也是一个组件，所以文件夹内的文件职责与上面的相同。但它也有一些特殊的地方。

  1. 这个是顶层的组件，一个Page就相当于对应了一整个html
  2. 它的css负责调整组件间的相互关系。推荐每个组件加一个class 命名为page，这可以在更高层的css中做处理用

甚至可以在pages和components中抽象出一个layouts层，它也是一个组件，如果两个页面非常像，比如说成绩查询和成绩预约页面，所需要的组件都是相同的，只是小部分文案不同，可以独立出一个layout，然后page渲染layout，layout渲染component

数据绑定

现在很多库推崇的都是不要直接操作DOM，而是操作虚拟DOM，然后由库来决定真实DOM要不要重新渲染。常见形式有单向绑定和双向绑定。

  • 单向绑定: 可以通过改变数据改变DOM，不可以通过改变DOM改变数据。react就是这种方式。整体感觉就是如果想改变DOM，那么就去改变数据。如果数据不变，DOM其实是也不能改变的。这种方式效率高，模型简单。我们只需要考虑数据的改变就好了，但实际使用中form的input会有个小坑（form中的input一般都绑定在一个数据上，如果输入，其实是改变了DOM，数据并没有改变。所以表现出来的现象是不能输入和删除）。
  • 双向绑定：通过改变数据改变DOM，也可以通过改变DOM改变数据。angular采用这种方式。这种方式其实写小应用挺好用，数据和DOM相当于是同一个东西，不会存在单向绑定中form的input的那种问题。但是效率低，并且如果应用复杂后，其实是引入了很多不确定性。
