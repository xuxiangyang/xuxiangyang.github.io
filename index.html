Feb 19th, 2016 4:26 pm

代码在 react_start_kit的basic_frame tag上

大致文件结构组织如下。

一个基本原则是这样的 不管是page还是component，本质上都是一个组件。一个组件是由这样一个文件夹组成的：（以Slogan为例）

  1. 文件夹名字与组件相同。所以文件夹名为Slogan
  2. 这个文件下有这个组件需要的所有东西，比如js、css，甚至是图片等。
  3. 文件夹下有一个Slogan.js与Slogan.css。js文件负责组件的显示，资源的整合。css负责组件内的布局，它不用关心组件间该如何排列。
  4. css 的className推荐以组件名作为前缀，并且一个组件的顶级className最好和组件名相同。

最基础的组件，放在components文件夹下。有几个要点

  1. 显示的内容最好不要写死，要通过props传递。比如文字内容，最好写进props，然后在Page中初始化。这样能最大程度的重复利用
  2. 组件要分的恰当，不要太大，也不要太小。基本原则是这样的：如果一个组件会在Page或其他Component中使用，那么就独立为一个文件夹。否则这是一个内部组件，放在相应组件即可。以Banner的显示为例
  3. 暴露在外面的是Banner这个组件，一个Banner由多个BannerImage组成，Banner负责控制BannerImage的初始化、轮播
  4. BannerImage寄生在Banner组件中，放在对应的js和css中。
  5. 如果以后有组件需要使用BannerImage，那么可以将它独立为一个文件夹，然后考虑换一个更通用的名字，例如ClickableImage等。

抽象出来了一类特殊的组件，叫做page。建议名字以Page结尾。它也是一个组件，所以文件夹内的文件职责与上面的相同。但它也有一些特殊的地方。

  1. 这个是顶层的组件，一个Page就相当于对应了一整个html
  2. 它的css负责调整组件间的相互关系。推荐每个组件加一个class 命名为page，这可以在更高层的css中做处理用

甚至可以在pages和components中抽象出一个layouts层，它也是一个组件，如果两个页面非常像，比如说成绩查询和成绩预约页面，所需要的组件都是相同的，只是小部分文案不同，可以独立出一个layout，然后page渲染layout，layout渲染component

2. React组件动起来

所谓动起来，其实就是state。通过需要state可以让页面DOM改变。 state是什么呢？所谓状态，其实一般就是受用户行为影响的、或者会随着时间改变的东西。 它和props的区别如下：

  1. props是父级传递下来的，在子级内不可以改变。state是子级内部的，是可以改变的。
  2. props是静态的，它一般用来构成html的“骨架”。父级可以将自己的state当作props传递给子级， 对子级而言，得到的props是静态的。

举个例子，form提交。

效果如 [Image: file:///-/blob/JIHAAAqTc9C/6K58Jv6gf78ZNVLQOuNbTg]但你会发现这个input不能输入，不能删除。这个原因是React的数据是单向绑定的。我们的输入其实是改变了input的value，但是并没有改变这个组件的state。所以这个value也不会改变。所以最常见的做法就是重写input的onChange函数，让它去修改这个组件的state 下面会用ref来选取到input，它的作用和css选择器相当。

这里面有几个知识点：

  1. this需要bind才能够访问到，js是把函数当作作用域的。
  2. 改变state，不能通过this.state.username=”ss”来改变需要通过setState改变。同样地props也不能通过this.props.somePorps = “value”来改变，要使用setProps来改变
  3. ref，相当于css选择器，用来定位到一个html元素
  4. setState只会修改传进去的hash中有的key，假设一个state是{a: 1, b: 2}，我们通过this.setState({b: 3})，这时，state是{a: 1, b: 3}。

0.现代前端基础常识

Feb 19th, 2016 4:21 pm

npm: 相当于ruby中的bundle，利用package.json管理

但它的功能更强大，除了包含作者等等基本信息，还用scripts定义了可以执行的脚本。dependencies是你的依赖，就像gem的依赖。devDependencies是你开发时用的工具，里面的东西和ruby中类似于pry等gem相当。几个常用的命令：

npm负责你程序的脚本，依赖、下载。所以看一个前端，第一个应该看它的package.json

webpack: 打包工具

它可以将我们各种js、css、图片。利用各种loader加载好。这样，我们需要任何一个资源（不管是js、css、图片）都可以使用require来加载了。 它依赖于webpack.config.js。里面会定义各种各样的loader和plugin。之所以可以require各种资源，是因为给他们定义了对应的loader。 它有一个webpack-dev-server可以用来做开发是起web服务

babel: 用来转义js，jsx、ecma6、ecma7都由它转义

js出了各种写法，比如jsx、ecma6、ecma7等等。但很多浏览器不支持这种写法，所以需要babel来进行转义，转义成浏览器支持的形式。babel分成了好多个项目。要注意分清楚。

JSX: 就是js、html混用，写法类似下面的

但jsx本身是不被浏览器支持的。他需要babel进行转义。 他有一些不同

  1. class属性变为了className，因为class与js语法冲突
  2. 如果往原生HTML中加入HTML规范中不存在的属性，需要使用data-前缀

这个东西可以用sublime package Babel高亮。每个项目下有一个m.sublime-project。以后的输入用subl m.sublime-project代替subl . 会更合理。直接使用这个项目的特定sublime配置

ECMA6/ECMA7: 这是js的新规范。里面定义了一些新的语法。同样目前不能被所有浏览器直接执行。需要用babel进行转义。

数据绑定

现在很多库推崇的都是不要直接操作DOM，而是操作虚拟DOM，然后由库来决定真实DOM要不要重新渲染。常见形式有单向绑定和双向绑定。

  • 单向绑定: 可以通过改变数据改变DOM，不可以通过改变DOM改变数据。react就是这种方式。整体感觉就是如果想改变DOM，那么就去改变数据。如果数据不变，DOM其实是也不能改变的。这种方式效率高，模型简单。我们只需要考虑数据的改变就好了，但实际使用中form的input会有个小坑（form中的input一般都绑定在一个数据上，如果输入，其实是改变了DOM，数据并没有改变。所以表现出来的现象是不能输入和删除）。
  • 双向绑定：通过改变数据改变DOM，也可以通过改变DOM改变数据。angular采用这种方式。这种方式其实写小应用挺好用，数据和DOM相当于是同一个东西，不会存在单向绑定中form的input的那种问题。但是效率低，并且如果应用复杂后，其实是引入了很多不确定性。

依赖注入

angular中着重讲的概念。其实就是函数式编程。在js中函数也是对象，函数可以和其他对象一样可以当参数传递。所以会见到很多这种写法

使用的时候就是把声明好的函数传进去。在angular中，一般用这种方式把http、route等函数传进去。同样react也会用这种方式，传递一些函数。比如flux/redux中的dispatch、getState等

Golang 黑魔法之一: 使用strings.Split()函数时，如果string不包含sep则返回[string]而不是[]

Feb 16th, 2015 10:43 am

学习golang一小段时间了，深深被其敢于改变的特性所吸引。但有时也不免改动过大，让人难以理解

这是一个随手笔记，可能会有错误。。。。。

直接上代码

package main

import (
    "fmt"
    "strings"
)

func main() {
    fmt.Println(strings.Split("", "ss"))
    fmt.Println(strings.Split("a", "ss"))
}

它将有如下输出

[""]
["a"]

所以在golang中认为split失败时需要返回一个Array，这个Array的元素是其本身。其他语言(至少在我的认识中)则认为返回一个空Array。这是对我而言一个坑。因为如果这样使用，会有问题

package main

import (
    "fmt"
    "strings"
)

func main() {
    fmt.Println(strings.Split("", "ss"))
    for _, word := range strings.Split("a", "ss") {
        fmt.Println(word)
    }
}

我希望它是输入(所有的双引号只是用来区分字符串长度什么的，真正的输出是不会有双引号的)

0
"" //空字符串

但它的输出会是这个样子

1
"a"
